Scope

This alpha flow validates the current PySide6 desktop server and client roles as shipped by the Linux desktop artifacts. It covers install, first unlock, TALON-owned Reticulum setup, client enrollment, two-way sync, operational workflows, server-only administration, client-side restrictions, and restart behavior.

Server Tester

  • Installs and launches the server artifact.
  • Creates the first local encrypted server database.
  • Configures Reticulum with transport enabled.
  • Generates one-time client enrollment tokens.
  • Exercises server-only controls: enrollment, clients, audit, keys, mission lifecycle, document upload/delete, record delete, lease renewal, and revocation.

Client Tester

  • Installs and launches the client artifact.
  • Creates the first local encrypted client database.
  • Configures Reticulum to reach the server.
  • Enrolls with a full TOKEN:SERVER_HASH value.
  • Exercises field workflows: map, SITREPs, assets, missions, assignments, incidents, chat, documents, and operator profile.
Use test data only. Do not use live incident information, production RNS identities, real private documents, or a passphrase reused anywhere else.

Test Bed

Recommended Layout

  • Use two Linux machines, two VMs, or two isolated Linux user accounts: one server role and one client role.
  • Avoid installing both role artifacts into the same user profile unless the test is specifically about destructive role switching. The installer protects role switches and requires the exact phrase DELETE TALON DATA before deleting previous TALON files.
  • Use a reachable Reticulum path. For a controlled LAN alpha test, TCP Server on the server and TCP Client on the client are deterministic. A direct TCP privacy warning is expected when this path is used.
  • AutoInterface, Yggdrasil Server/Client, and i2pd Server/Client are also available from the Network Setup dialog.

End-To-End Flow

A. Install And Smoke Test

IDRoleActionExpected ResultEvidence
A1Server Extract and install the server artifact:
tar -xzf talon-desktop-server-linux.tar.gz
cd talon-desktop-server-linux
bash ./install.sh --yes --smoke-test
 Installer creates talon-desktop-server, T.A.L.O.N. Server, ~/.talon-server/talon.ini, and ~/.talon-server/reticulum/config. Smoke output includes TALON_DESKTOP_SMOKE_OK server. Installer output, package filename, screenshot or terminal capture.
A2Client Extract and install the client artifact:
tar -xzf talon-desktop-client-linux.tar.gz
cd talon-desktop-client-linux
bash ./install.sh --yes --smoke-test
 Installer creates talon-desktop-client, T.A.L.O.N. Client, ~/.talon/talon.ini, and ~/.talon/reticulum/config. Smoke output includes TALON_DESKTOP_SMOKE_OK client. Installer output, package filename, screenshot or terminal capture.
A3Both Launch each role from its role-specific launcher. The login window title shows the correct role: T.A.L.O.N. Desktop [SERVER] or T.A.L.O.N. Desktop [CLIENT]. Screenshot of each login window.

B. First Unlock And Network Setup

IDRoleActionExpected ResultEvidence
B1Server Enter a strong local passphrase and select Unlock. In Network Setup, select TCP Server or the agreed deployment template, validate, and save. Network Setup is available only after unlock. The Reticulum config validates. Server mode uses enable_transport = True. The main window opens with all common pages plus Enrollment, Clients, Audit, and Keys. Screenshot of Network Setup validation and dashboard.
B2Client Enter a strong local passphrase and select Unlock. In Network Setup, select TCP Client or the agreed deployment template, enter the server address/port if prompted, validate, and save. Client mode uses enable_transport = False. After network startup, a fresh client shows the Client Enrollment group with token and callsign fields instead of the main window. Screenshot of Network Setup validation and enrollment prompt.
B3Both If TCP is used, acknowledge any TCP privacy warning after network startup. Warning appears for direct TCP and explains that TCP can expose addressing metadata. This is a pass condition for a TCP alpha test. Screenshot or note that warning appeared.

C. Client Enrollment

IDRoleActionExpected ResultEvidence
C1Server Open Enrollment, select Generate Token, then Copy. A one-time token is generated. The copied value is the combined TOKEN:SERVER_HASH value when the server hash is initialized. Pending token count increases until the client consumes it. Screenshot with token preview only, not the full token in public reports.
C2Client Paste the full token into the enrollment prompt, enter a unique callsign such as ALPHA1, and select Enroll Client. Client enrollment succeeds and the main window opens. If it fails, the login window shows Enrollment failed: followed by the core error. Client dashboard screenshot.
C3Server Open Clients and Operators. The enrolled callsign appears as active, not revoked. The consumed token no longer appears as pending. Server Clients screenshot and pending token count.

D. Server-Origin Data Sync

IDRoleActionExpected ResultEvidence
D1Server Open Assets, select New, create Alpha Cache 01 as category cache, and add a test latitude/longitude. The asset appears in the server Assets table. The dashboard asset count increases. Server Assets screenshot.
D2Client Refresh Assets and Map. Alpha Cache 01 appears on the client. The Map page shows it in overlays and the asset filter can hide/show it. Client Assets and Map screenshots.
D3Server Open Documents, upload a small allowed file such as alpha-brief.txt, and enter a description. Upload succeeds. Filename, size, uploader, timestamp, and SHA-256 preview appear. Upload is server-only. Server Documents screenshot.
D4Client Open Documents, refresh, select the uploaded file, and download/save it. The document metadata syncs to the client. Download succeeds through the active sync session and integrity check passes. Client Documents screenshot and saved file hash if collected.

E. Client-Origin Data Sync

IDRoleActionExpected ResultEvidence
E1Client Create an asset named Client Rally 01 with category rally_point and a test location. The client asset appears locally. If the client is temporarily disconnected, the Dashboard pending outbox count should increase. Client Assets screenshot and dashboard count.
E2Server Refresh Assets. Client Rally 01 syncs to the server. Server can verify or unverify the asset. Server Assets screenshot.
E3Client Open SITREPs, select a template, set level PRIORITY, attach the client asset or coordinates, enter a body, and send. SITREP appears in the client feed with author callsign, level, status, location/link data, and no audio unless the level is FLASH or FLASH_OVERRIDE and Audio is enabled. Client SITREP screenshot.
E4Server Refresh SITREPs, select the client SITREP, then test Acknowledge, Assign, Close, and Create Incident. Follow-ups appear in detail. Status and assignment fields update. Linked incident opens in the Incidents page. Server SITREP detail and linked Incident screenshots.

F. Mission, Assignment, Incident, And Chat Workflows

IDRoleActionExpected ResultEvidence
F1Client Open Missions, select New, create a mission request with title, priority, requested asset, AO polygon, route, timing, objectives, constraints, and support notes. Client-created mission is submitted as pending approval. Geometry validation rejects an AO polygon with fewer than three points. Client Missions screenshot and any validation notes.
F2Server Open Missions, select the pending mission, approve it, and review asset allocation. Then complete or abort a duplicate test mission. Approval changes mission status to active, creates/updates the mission chat channel, and links allocated assets. Server lifecycle buttons include approve, reject, abort, complete, and delete according to mission status. Server approval dialog and active mission screenshot.
F3Either Open Assignments, create an assignment, add a location, then have the client submit a Check-In. Assignment appears on both roles. Check-in appears in recent activity. Server can acknowledge the check-in. Assignments board screenshots from both roles.
F4Either Open Incidents, create a report linked to the assignment, asset, mission, or SITREP, and mark follow-up needed. Incident appears with category, severity, title, location, narrative, actions, outcome, and follow-up filters. Incident list/detail screenshot.
F5Either Open Chat, create a custom channel, send a normal message, send an urgent message, attach a grid reference, and open a direct-message channel with another operator if available. Messages sync both ways. Urgent messages appear in alert surfaces. Grid references are built from known map positions such as assets, zones, waypoints, and located SITREPs. DM channels are server-readable in the current core. Chat screenshots from both roles.

IDTestExpected ResultEvidence
P1Client navigation after enrollment.Client has Dashboard, Map, SITREPs, Assets, Missions, Assignments, Incidents, Chat, Documents, and Operators. Client does not show Enrollment, Clients, Audit, or Keys.Client nav screenshot.
P2Server navigation after unlock.Server has all client pages plus Enrollment, Clients, Audit, and Keys.Server nav screenshot.
P3Client tries to upload or delete a document.Upload is unavailable or reports Upload is available in server mode only. Delete is unavailable in client mode.Client Documents screenshot.
P4Client tries to delete a chat channel/message.Delete controls are unavailable or disabled. Server can delete eligible non-default channels/messages.Client and server Chat screenshots.
P5Client tries to verify its own asset.Client cannot verify an asset created by the same local operator. Server can verify it.Asset detail screenshots.
P6Server uploads blocked file types such as test.sh or oversized files over 50 MB.Upload fails with an operator-facing document error. Allowed alpha uploads include PDF, TXT, MD, CSV, JSON, GeoJSON, and common image formats.Document error text.
P7Fresh database unlock with weak passphrase such as a short phrase or predictable pattern.Unlock fails with passphrase policy text. Repeated failures trigger temporary unlock delay.Error text and timestamp.
P8Keys page Rotate Group Key.Button reports that group key rotation is not implemented yet. It must not pretend to rotate keys.Dialog screenshot.

Reliability Tests

IDActionExpected ResultEvidence
R1 Close and reopen both roles. Unlock with the same local passphrases. Server opens directly to the main window after network setup. Enrolled client opens to the main window without asking for enrollment again. Existing records persist. Dashboard screenshots after restart.
R2 Disconnect the client network after enrollment. Create one asset and one SITREP. Reconnect and wait for sync. Client can create offline-creatable records. Pending outbox count increases while disconnected, then decreases after reconnect and server receipt. Dashboard counts before and after reconnect.
R3 Restart only the server while the client remains open. Client should recover when the server returns. New client records should sync after the server is reachable again. Client status, server status, launcher logs.
R4 Use a spare enrolled client, then have the server revoke that operator from Clients or Keys. Spare client shows a modal locked state and cannot continue normal operation. Server audit can be filtered for the revocation event. Client lock screenshot and Audit screenshot.
R5 Have the server renew the primary client's lease. Operator row lease status updates. Audit severity for lease renewal is positive. Any lease lock should close after renewal. Clients and Audit screenshots.

Exit Criteria

Minimum alpha pass: both artifacts install and smoke test, one client enrolls, server-origin and client-origin records sync, core operational pages are usable, server-only permissions hold, restart persistence works, and no data integrity or lockout defect is observed.

AreaPass Requirement
InstallRole-specific launchers and desktop entries are correct. No destructive role switch is needed for the normal two-role test bed.
Unlock and networkStrong passphrases work, weak passphrases fail, Reticulum config validates, and TCP warnings appear when TCP is active.
EnrollmentOne-time token is generated, consumed, and removed from pending tokens. Client receives an operator identity and reaches the main window.
SyncAssets, SITREPs, missions, assignments, incidents, chat, and document metadata behave according to role and sync expectations.
PermissionsClient cannot access server-only admin pages or server-only upload/delete/lifecycle controls.
RecoveryReopening apps preserves data and enrollment. Temporary network loss does not corrupt records.

Bug Report Packet

For every failure, attach the following:

  • Step ID, role, expected result, and actual result.
  • Package filename and OS version.
  • Network method and whether TCP warning appeared.
  • Screenshot of the affected page or dialog.
  • Relevant launcher log from ${XDG_STATE_HOME:-$HOME/.local/state}/talon.
  • Whether the problem reproduces after app restart.
